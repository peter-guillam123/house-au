House UK · AU · US

Under the hood

Three Houses, one shape

House is a search tool for parliamentary records, built for journalists at the Guardian. One search box across spoken contributions, bills, votes, hearings — or the equivalent in each country; a chart that plots how a topic has been discussed over time; a small region tag in the corner because there are now three editions: UK Parliament, Australia’s federal parliament, and the US Congress.

All three editions look the same on purpose. The point of this document is to show what’s not the same — the plumbing underneath. Three editions, three different relationships with the data each parliament publishes, three different cost shapes. For a non-technical reader curious about why the same search box can hide three genuinely different systems, and what those differences mean for the journalism the tool is there to support.

The same shop window

All three versions look the same on purpose

A journalist using one shouldn’t have to learn a new interface to use the others. So all three editions ship the same surfaces in the same shape, the same typography, the same filter vocabulary, the same keyboard hints. The region tag in the masthead is the only obvious mark of difference. The number of surfaces differs — not for design reasons but because each parliament’s data lets you do different things.

UK

House · UK

4 surfaces

Search, Deep Dive, Committees, Transcripts. Six years of select-committee oral evidence indexed for full-text search.

AU

House · AU

3 surfaces

Search, Deep Dive, Estimates. The committee-transcripts surface UK has doesn’t apply — AU’s editorially equivalent material lives in Senate Estimates, which gets its own tab.

US

House · US

2 surfaces

Search (with the Deep Dive chart folded in as a filter), and Lobbying. The headline US surface is Lobbying Disclosure Act filings — quarterly LD-2 reports, searchable on registrant, client, lobbyist, issue and (the gold seam) the free-text issue description.

The fundamental split

One fetches live. One publishes nightly. One does both.

The UK Parliament runs a public, CORS-friendly API: hit it with a search term, get back ranked results in under a second. The Australian Parliament doesn’t, so AU inverts the model and harvests the whole corpus on a schedule, ships it as static JSON. US is a hybrid: GovInfo is live but doesn’t send CORS headers (so a tiny proxy is necessary), and Lobbying Disclosure filings are quarterly batches with token auth (so they get the AU treatment). Three editions, three points on a spectrum.

Live request, per search Built ahead of time, served as static files

UK leans on Parliament’s API. A search round-trips through a tiny Cloudflare Worker (which only exists to add CORS headers and dodge browser-cache edge cases) and out to four parliamentary endpoints in parallel. Round-trip in the order of half a second.

A search through three pipes

What happens when you search a term

The same query, the same result list shape, three different paths through. UK is shortest at runtime. AU is shortest at every other moment of the day, because nothing’s happening on a server while the user waits. US splits the difference: fast for daily-news searches, instant for Lobbying (it’s already on disk).

UK · Live API

  1. Browser hits the Worker

    Search form posts to the Cloudflare Worker URL with the query and filters.

  2. Worker fans out

    Four upstream API calls go out in parallel: Hansard, Questions & Statements, Members, Committees.

  3. Each API ranks and returns

    Parliament’s search engines do the heavy lifting. Each returns its own ranked top 20.

  4. Worker proxies the responses

    Adds CORS headers, optionally caches at the edge for 5 min, returns to the browser.

  5. Browser merges and renders

    UI deduplicates across the four streams, sorts by date, builds the result list. ~500ms total.

AU · Pre-published shards

  1. Browser fetches the manifest

    A tiny hansard-archives.json describes what shards exist and which dates they cover.

  2. Pick the shards in range

    Filter the date range against the manifest. A “Last 5 years” search picks ~17 month-shards plus the 90-day rolling shard.

  3. Fetch in parallel batches

    Four shards at a time, decompressed on the fly via the browser’s built-in DecompressionStream when they’re gzipped archives.

  4. Filter in JavaScript

    The full text of every contribution is right there in memory; a regex pass finds matches. Members get party-stamped from members.json on the same fetch path.

  5. Browser builds the result list

    Same renderer as UK, same shape. ~1–3 seconds total depending on date range.

US · Hybrid

  1. Search hits the Worker (live)

    POST to the Cloudflare Worker, which adds the api.data.gov key and forwards to GovInfo’s search endpoint. GovInfo lacks CORS, so the Worker is mandatory, not optional.

  2. Chart fans out monthly counts

    In parallel, the chart fires 12–60 count-only queries (one per month) to draw the bar chart on top of the result list. Worker edge-cache makes repeats instant.

  3. Snippets fetch in the background

    GovInfo Search returns titles only; for each visible result the browser fetches the granule HTML (still through the Worker) and extracts a snippet around the match. Bounded concurrency 4.

  4. Lobbying reads static shards (built)

    Different surface, AU-shaped pipe: the browser fetches the relevant quarterly LD-2 shard from GitHub Pages, decompresses gzip, runs the search in memory. ~5MB per quarter.

  5. Quarterly action keeps shards fresh

    A GitHub Action harvests new LD-2 filings from lda.senate.gov each time a quarter’s filings drop (~20 days after quarter end), commits the shard back to the repo.

The cost shifts in time, not size. UK pays per-search. AU pays once a day at 18:00 UTC, then nothing. US pays per-search for the live half and once a quarter for the static half.

Why it matters

Plumbing shapes editorial possibility

The plumbing isn’t neutral. It quietly determines what each version can offer.

UK can afford to expose more surfaces because the cost is per-API-call: adding a new search type just means a new endpoint to call. Want to filter by select committee? The Committees API can do that. Want to count how often a Lord has spoken on Brexit? The Members API knows who they are; Hansard knows who said what. The plumbing scales to whatever Parliament publishes.

AU has to be more selective. Every surface is a harvester to maintain, a shard format to keep stable, a chunk of GitHub Pages bandwidth on the wire. So three things done well rather than five things done approximately: Search across Hansard, Deep Dive for term-over-time, and Estimates as the headline accountability surface. The committee-transcripts feature UK has would’ve required a third pipeline of similar size to the Hansard one. Skipped, on purpose.

US reads the room differently again. GovInfo gives us a huge live corpus (Congressional Record back to 1995, bills, hearings, Federal Register, FEC) but no speaker-level metadata, so a UK-style Most-Contributions leaderboard isn’t feasible without paying a 10k-fetch harvest cost. The Deep Dive chart folds into Search instead of being its own page. Meanwhile the Lobbying Disclosure Act produces something neither UK nor AU has — a quarterly, structured, dollar-attached record of who lobbied whom about what — so that becomes the US-distinctive surface, AU-shaped under the hood.

All three versions are deliberate. UK’s thinness is because it can be; AU’s fatness is because it has to be; US is hybrid because the sources are too. Same editorial product through three different plumbings.

A note for anyone porting this

If you want to do this for another parliament

UK, AU and US bookend the cost-shape spectrum and fill in its middle. Most parliaments will land somewhere along it. The thing that travels well is the editorial shape: a handful of search surfaces, restrained editorial typography, one accent colour from the country’s civic palette, mono-caps eyebrows, hairline rules. The data plumbing follows from what’s actually available.

The honest test for a new edition: is there an accountability surface that runs on a regular calendar, produces dense journalistic material, and isn’t already covered by the chamber-debate search? UK has Committees and Transcripts. AU has Senate Estimates. US has Lobbying Disclosure. If your target parliament has one, build it as its own surface. If not, two or three surfaces is enough.
