House UK · AU

Under the hood

Two Houses, one shape

House is a search tool for parliamentary records, built for journalists at the Guardian. One search box across spoken contributions, written questions, ministerial statements and committee material; a “Deep Dive” view that plots how a topic has been discussed over time; a small region tag in the corner because there are now two editions — one for the UK Parliament, one for Australia’s federal parliament.

Both editions look the same on purpose. The point of this document is to show what’s not the same — the plumbing underneath. For a non-technical reader curious about why the same search box can hide two genuinely different systems, and what those differences mean for the journalism the tool is there to support. We’ll engage with the technicalities where it matters, but the goal is to make the shape of each system legible without a computer-science background.

The same shop window

Both versions look the same on purpose

A journalist using one shouldn’t have to learn a new interface to use the other. So both editions ship the same five surfaces, the same typography, the same filter shapes, the same keyboard hints. The region tag in the masthead is the only obvious mark of difference.

The fundamental split

One fetches at search time. The other publishes daily.

The UK Parliament runs a public, CORS-friendly API: hit it with a search term, get back ranked results in under a second. The Australian Parliament doesn’t. So the AU version inverts the model — it harvests the whole corpus on a schedule and ships it as static JSON. The user never knows.

Live request, per search Built daily, served as static files

UK leans on parliament’s API. A search round-trips through a tiny Cloudflare Worker (which only exists to add CORS headers and dodge browser-cache edge cases) and out to four parliamentary endpoints in parallel. Round-trip in the order of half a second.

A search through both pipes

What happens when you type “AUKUS”

The same query, the same result list shape, two opposite paths. UK’s is shorter at runtime; AU’s is shorter at every other moment of the day, because nothing’s happening on a server that the user is waiting for.

UK · Live API

  1. Browser hits the Worker

    Search form posts to the Cloudflare Worker URL with the query and filters.

  2. Worker fans out

    Four upstream API calls go out in parallel: Hansard, Questions & Statements, Members, Committees.

  3. Each API ranks and returns

    Parliament’s search engines do the heavy lifting. Each returns its own ranked top 20.

  4. Worker proxies the responses

    Adds CORS headers, optionally caches at the edge for 5 min, returns to the browser.

  5. Browser merges and renders

    UI deduplicates across the four streams, sorts by date, builds the result list. ~500ms total.

AU · Pre-published shards

  1. Browser fetches the manifest

    A tiny hansard-archives.json describes what shards exist and which dates they cover.

  2. Pick the shards in range

    Filter the date range against the manifest. A “Last 5 years” search picks ~17 month-shards plus the 90-day rolling shard.

  3. Fetch in parallel batches

    Four shards at a time, decompressed on the fly via the browser’s built-in DecompressionStream when they’re gzipped archives.

  4. Filter in JavaScript

    The full text of every contribution is right there in memory; a regex pass finds matches. Members get party-stamped from members.json on the same fetch path.

  5. Browser builds the result list

    Same renderer as UK, same shape. ~1–3 seconds total depending on date range.

The cost shifts in time, not size. UK pays per-search; AU pays once a day at 18:00 UTC and then nothing until the following night.

What’s actually different

Same UI, very different mass underneath

UK AU
Source data Live, public APIs from Parliament. Five of them, with full-text search built in. No public API. The website’s own JSON endpoint, used internally by aph.gov.au, fetched politely by a nightly job.
Where data lives Parliament’s servers. We never copy it. In the repo. ~33 shard files, ~2GB on disk, served from GitHub Pages. Tiered: per-month for the live two years, per-quarter gzipped for the older archive.
Run-time cost One Cloudflare Worker, free-tier (100k req/day). 4 upstream API calls per search. Zero compute. Static files served from CDN. Browser does all the search work.
Build-time cost None ongoing. The transcripts archive runs a daily 90-day-window job (~10 minutes). Nightly GitHub Action runs the whole pipeline: discover sitting days, fetch new fragments, parse, rebuild shards, refresh members, repeat for Senate Estimates. ~5 min on a normal night.
Freshness Roughly real-time — whatever Parliament published, you can search. ~24 hours behind, because Hansard publishes ~24h after a sitting and we harvest after that.
Member metadata Members API returns party, constituency, role per query. No equivalent endpoint. Each MP’s profile page is scraped once, cached as members.json, joined to contributions by their stable parliamentarian ID.
Notable AU-only surface — Senate Estimates: twice-yearly hearings where senators question departmental officials. The single most journalistically dense AU accountability surface; UK has no equivalent. Gets its own tab with round / committee / portfolio filters.
Notable UK-only surface Transcripts: full-text search of select-committee oral evidence. UK’s Committees API exposes the words; AU’s site doesn’t and the underlying ParlInfo XML sits behind an Azure WAF that blocks bot traffic. Skipped for AU. —

Why it matters

Plumbing shapes editorial possibility

The plumbing isn’t neutral. It quietly determines what each version can offer.

UK can afford to expose more surfaces because the cost is per-API-call: adding a new search type just means a new endpoint to call. Want to filter by select committee? The Committees API can do that. Want to count how often a Lord has spoken on Brexit? The Members API knows who they are; Hansard knows who said what. The plumbing scales to whatever Parliament publishes.

AU has to be more selective. Every surface is a harvester to maintain, a shard format to keep stable, a chunk of GitHub Pages bandwidth on the wire. So we’ve picked three things to do well rather than five things to do approximately: Search across Hansard, Deep Dive for term-over-time, and Estimates as the headline accountability surface. The committee-transcripts feature UK has would’ve required a third pipeline of similar size to the Hansard one, fighting Parliament’s site at every step. Skipped, on purpose.

Both versions are deliberate, in other words. UK’s thinness is because it can be; AU’s fatness is because it has to be. Same editorial product through opposite plumbing.

A note for anyone porting this

If you want to do this for another parliament

The UK and AU patterns bookend the cost-shape spectrum. Most parliaments will land somewhere in between — some live API, some scraping, some static publication. The thing that travels well is the editorial shape: five (or four, or three) search surfaces, restrained editorial typography, one accent colour from the country’s civic palette, mono-caps eyebrows, hairline rules. The data plumbing follows from what’s actually available.

The honest test for a third edition: is there a Senate Estimates analogue? An accountability surface that runs on a regular calendar, produces dense journalistic material, and isn’t already covered by the chamber-debate search. If yes, build it as its own surface. If no, three surfaces is enough.
